# 项目亮点与局限性

## 项目亮点

### 1. 不是普通 RAG 问答

普通 RAG Demo 通常只做三件事：

1. 上传文档。
2. 检索片段。
3. 调用 LLM 回答。

本项目额外做了：

- 风险识别。
- 制度冲突识别。
- 证据引用。
- 结构化报告。
- 工作流 Trace。
- 人工审批。
- 权限隔离。
- 评测和运维诊断。

这更接近企业内部合规、法务、售前和客服知识库的真实要求。

### 2. 上传阶段就写入向量数据库

文档上传后会立即：

- 解析文本。
- 切片。
- 生成 embedding。
- 写入 `document_chunks.embedding`。

提问时只需要对 query 生成 embedding，然后走 pgvector 相似度检索。这是更接近生产系统的做法。

### 3. 混合检索和重排

检索不是只靠向量：

- 关键词检索适合命中制度词、审批词、风险词。
- 向量检索适合语义相似问题。
- 融合排序平衡两类信号。
- Reranker 对候选证据重新排序。

Trace 中可以看到候选排序过程，便于解释“为什么引用这条证据”。

### 4. 多 Agent 工作流

项目使用 LangGraph 组织流程：

```text
检索 Agent -> 审计 Agent -> 报告 Agent -> 人工确认节点
```

这样做的价值：

- 每一步职责清楚。
- 每一步都能记录 Trace。
- 审计逻辑和报告生成解耦。
- 高风险结果可以插入人工审批。
- 后续可以替换任意节点，例如把规则审计替换成 LLM 审计。

### 5. 证据优先

系统强制要求：

- 没有检索证据就不回答。
- 回答必须带引用。
- 风险发现必须绑定证据。
- 引用包含来源位置。

这比“LLM 看起来回答得很流畅”更适合企业场景。

### 6. 可观测性

每次审计会记录：

- prompt
- tool calls
- duration
- token 估算
- failure reason
- LLM 状态
- 检索候选排序

这能回答面试官常问的问题：

```text
线上 Agent 出错了，你怎么排查？
```

### 7. 权限隔离

项目实现了演示级权限控制：

- Alice 是 editor。
- Bob 是 viewer。
- 不同用户只能访问自己有权限的知识库。
- viewer 不能上传，也不能访问系统诊断。

这覆盖了企业知识库常见的安全要求。

### 8. 可评测

项目内置 60 条评测问题，并输出：

- Recall@1
- Recall@3
- Citation accuracy
- Answer quality
- Risk type accuracy
- Conflict accuracy
- Evidence binding accuracy
- Review trigger accuracy

这能说明项目不是只靠主观演示，而是有可复现指标。

### 9. 可部署和可运维

Docker Compose 一键启动：

- FastAPI
- PostgreSQL + pgvector
- MinIO

系统状态面板能展示：

- 数据库状态。
- pgvector 状态。
- 文档和切片数量。
- 索引异常。
- 当前模型配置。

## 当前局限性

### 1. 权限模型仍是演示级

当前权限模型足够展示用户和知识库隔离，但还不是完整生产级 IAM。后续可以增加：

- 租户管理。
- 部门树。
- 文档级 ACL。
- 角色继承。
- 管理员后台。

### 2. 审计规则主要是规则驱动

当前审计 Agent 使用规则识别风险和冲突，优点是稳定、可测，缺点是泛化能力有限。后续可以加入：

- LLM 审计归纳。
- 可配置规则引擎。
- 人工标注反馈闭环。

### 3. OCR 和复杂版面解析还可以增强

扫描 PDF 已支持 OCR，但复杂合同、跨页表格、图片中的表格还可以继续优化：

- 版面分析。
- 表格结构还原。
- OCR 置信度。
- 页面质量诊断。

### 4. 评测集规模还可以扩大

当前是 60 条用例，适合项目展示。生产环境应该扩展到：

- 更多文档类型。
- 更多业务问题。
- 更多负样本。
- 多人标注。
- LLM-as-judge + 人工复核。

### 5. 前端是演示管理台

前端目前重点是展示 Agent 能力，不是完整后台系统。后续可以增加：

- 知识库管理。
- 用户和角色管理。
- 文档删除和重建索引。
- 审计任务队列。
- 评测任务管理。
