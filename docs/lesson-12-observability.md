# 第十二课：工作流可观测性

本课把“能跑”升级成“能查”。每次问答都会返回一条可追踪的工作流记录，方便排查检索失败、规则判断错误和导出问题。

## 新增内容

每次 `/api/ask` 和 `/api/reports/export` 现在都会返回或记录：

- `trace_id`
- `workflow_trace`
- 每一步的 `name`
- 每一步的 `status`
- 每一步的 `detail`
- 每一步的 `duration_ms`
- 每一步的 `prompt`
- 每一步的 `tool_calls`
- 每一步的 `input_tokens`
- 每一步的 `output_tokens`
- 每一步的 `failure_reason`

## 为什么要记录 prompt

在真实 Agent 系统里，问题通常不是“有没有回答”，而是：

- 输入给每一步的内容是什么
- 工具调用顺序是否正确
- 哪一步超时了
- 哪一步因为空检索或规则判断失败

所以工作流必须留下可读、可回放的轨迹。

## token 是怎么来的

当前项目还没有接真实大模型，所以这里先使用估算值：

- 按文本长度粗略换算输入 token
- 按证据数、发现数粗略估算输出 token

这不是生产级计费口径，但足够做学习和面试展示。

## 页面如何展示

结果区现在多了一个“工作流追踪”面板，能直接看到：

1. 检索 Agent
2. 审计 Agent
3. 报告 Agent

每一步都带有耗时和失败原因，方便在演示时解释系统是怎么得出结论的。
